Outputs
=======

This section describes the output files generated by Osprey. For details on how to specify the output folder in your job file, please see the :doc:`jobfiles` section of the documentation.

Depending on the options specified in the job file, Osprey generates a variety of output files. We usually give our output folder the name ``derivatives`` to be consistent with the `BIDS standard <https://bids.neuroimaging.io/>`_, but you can specify any name you like in your job file. If you look into your output folder, you may see any of the following folders and files:


.. code-block:: bash

    derivatives/
    └── DefacedNII/
    └── FilesSPM/
    └── jMRUIFiles/
    └── LCModelFiles/
    └── NIfTIMRS/
    └── QuantifyResults/
    └── RDAFiles/
    └── SDATFiles/
    └── SegMaps/
    └── VoxelMasks/
    jobSDAT.mat
    LogFile.txt
    QM_processed_spectra.json
    QM_processed_spectra.tsv
    subject_names_and_excluded.json
    subject_names_and_excluded.tsv
    SummaryMRSinMRS.md

The following sections describe the content of these folders and files in more detail.

MRS Container (.mat file)
-------------------------

The ``.mat`` file with the same name as your job file (e.g., ``jobSDAT.mat``) contains a MATLAB structure variable named ``MRSCont``. This variable is the main container for all MRS data and metadata in Osprey, and it is used as the main carrier for all processing and quantification steps. It contains the raw MRS data, the processed MRS data, quantification results, and much more. You can directly re-load this file into your MATLAB workspace using the following command:

.. code-block:: matlab
    
    MRSCont = load('path/to/your/output/folder/jobSDAT.mat');

LogFile.txt
-----------

The ``LogFile.txt`` file contains a log of all processing and quantification steps performed by Osprey, including the date and time of each step, the options used for each step, and any warnings or errors that occurred during processing. This file can be useful for troubleshooting and for keeping a record of your analysis.

SummaryMRSinMRS.md
------------------

The SummaryMRSinMRS.md file contains a quasi-standardized overview of the MRS methods in a human-readable (Markdown) format, according to the `highly-cited consensus on Minimum Reporting Standards in MRS (MRSinMRS) by Alexander Lin et al <https://pubmed.ncbi.nlm.nih.gov/33559967/>`__ . 

The MRSinMRS table contains a standardized description of the MRS hardware, data acquisition, analysis, and quality assessment that were used for acquiring and analyzing a particular MRS dataset. 

.. note ::

    Osprey attempts to fill in as many fields as possible from the available header information, but some fields may require manual input. If you notice any missing or incorrect information in the MRSinMRS table, please edit the SummaryMRSinMRS.md file manually to add or correct the information. 
    
    We also welcome contributions to improve the automatic filling of the MRSinMRS table in Osprey, so please feel free to `open a GitHub issue <https://github.com/schorschinho/osprey/issues>`_ or `submit a pull request <https://github.com/schorschinho/osprey/pulls>`_ if you have suggestions for improvement.

DefacedNII
----------

If you have set the job file option ``opts.img.deface`` to ``1`` (by default, this is set to ``0``), Osprey will generate defaced NIfTI files for all anatomical images specified in the job file. The defacing is performed using the ``spm_deface`` function in SPM12.

The defaced NIfTI files are stored in the ``DefacedNII`` folder within your output folder.

FilesSPM
-------- 

The ``FilesSPM`` folder contains outputs generated by the SPM12 segmentation routines. This includes segmented tissue maps in ``.nii.gz`` (NIfTI) files prefixed with ``c1``, ``c2``, and ``c3`` (i.e., gray matter, white matter, cerebrospinal fluid), ``seg8.mat`` (SPM segmentation results), and normalization/transformation fields.

MRS files written in particular MRS formats
-------------------------------------------

Osprey can save processed MRS spectra in various formats that can be used for further analysis in other software packages. The following sections describe the different formats that Osprey can save processed spectra in. Naming conventions for these files are consistent across formats:

Conventional (non-edited) spectra will always be saved in files with the suffix ``_A``.

Sub-spectra from edited experiments will be saved in separate files with the suffixes ``_A``, ``_B``, ``_diff1`` and ``sum`` (as well as  ``_C``, ``_D``for, and ``diff2`` for four-step editing experiments) for the edit-ON, edit-OFF, and difference spectra, respectively.

Lineshape reference and short-TE water reference spectra will be saved in separate files with the suffixes ``_REF`` and ``_W``, respectively.

jMRUIFiles
~~~~~~~~~~

If the job file option ``opts.savejMRUI`` is set to ``1`` (by default, this is set to ``0``), this folder will be created and will contain processed spectra in jMRUI format (.txt), which can be used for further analysis in the jMRUI software. Each file will be identified with a ``_jMRUI`` suffix.

LCModelFiles
~~~~~~~~~~~~

If the job file option ``opts.saveLCModel`` is set to ``1`` (by default, this is set to ``0``), this folder will be created and will contain processed spectra in LCModel format (.RAW), which can be used for further analysis in the LCModel software. If Osprey is set up to perform fitting with the built-in LCModel executable, these files will be generated regardless of the ``opts.saveLCModel`` value. Each file will be identified with a ``_LCM`` suffix.

NIfTIMRS
~~~~~~~~

If the job file option ``opts.saveNII`` is set to ``1`` (by default, this is set to ``0``), this folder will be created and will contain processed spectra in NIfTI-MRS format (.nii.gz), which can be used for further analysis in other software packages that support NIfTI-MRS files (e.g., FSL-MRS, spant, etc.). There is no explicit additional NIfTI-specific suffix for these files.

SDATFiles or RDAFiles
~~~~~~~~~~~~~~~~~~~~~

If the job file option ``opts.saveVendor`` is set to ``1`` (by default, this is set to ``0``), this folder will be created and will contain processed spectra in Philips SDAT/SPAR (.SDAT./.SPAR) or Siemens RDA (.RDA) formats, which can be used for further analysis in other software packages that support those files. There is no explicit additional SDAT- or RDA-specific suffix for these files.

QuantifyResults
---------------

The ``QuantifyResults`` folder contains the results of the quantification step in Osprey and will be generated after ``OspreyQuantify`` is run. 

This includes the estimated metabolite concentrations, fit parameters, and quality metrics for each spectrum. The results are saved in JSON/TSV pairs for easier access and interoperability with statistical software packages.

SegMaps
-------

The ``SegMaps`` folder contains the segmented tissue maps generated by SPM12 (stored in the ``FilesSPM`` folder), masked with the binary mask representing the MRS voxel (stored in the ``VoxelMask`` folder). 

Furthermore, the ``SegMaps`` folder contains a BIDS-compliant TSV/JSON pair with the relative tissue volume fractions of each voxel, which is used for partial volume correction during water-scaled quantification.

VoxelMasks
----------

The ``VoxelMasks`` folder is created when ``OspreyCoreg`` is executed. It contains binary masks representing the MRS voxel, which are generated based on the voxel coordinates and dimensions specified in the MRS file headers. These masks are used for co-registration, partial volume correction, and other processing steps that require knowledge of the voxel location and size. They are given in scanner space, and therefore labeled with the suffix ``space-scanner_mask.nii.gz``.

If the helper function ``osp_plotVoxelOverlap`` is called, Osprey will further generate the following output files in ``VoxelMasks``: 1) a NIfTI file for each individual subject containing the voxel mask in SPM152 space (with the suffix ``_space-spm152_mask.nii.gz``), and 2) the Dice-Sorensen coefficient map in SPM152 space (with the suffix ``_space-spm152_mask_VoxelOverlap.nii.gz``).
